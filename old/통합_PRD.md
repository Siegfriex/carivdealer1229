## 1. 실행 요약

### 1.1 프로젝트 비전
ForwardMax는 국내 중고차 딜러와 해외 바이어를 연결하는 B2B/B2C 중고차 수출 중개 플랫폼입니다. 딜러에게는 부가세 환급(9.09%)과 자동화된 서류 처리를 제공하고, 해외 바이어에게는 검증된 차량과 투명한 거래 프로세스를 제공합니다.

### 1.2 핵심 가치 제안
- **딜러 관점**: 복잡한 서류 처리 자동화, 부가세 환급, 빠른 자금 회전
- **바이어 관점**: 검증된 차량, 투명한 거래, 글로벌 접근성
- **플랫폼 관점**: 문서 자동화 SaaS(AutoExport) + D2G 거래 플랫폼(Cariv) 통합 솔루션

### 1.3 시장 기회
- 연간 60만 대 이상, 6조원 규모의 중고차 수출 시장
- 디지털 전환율이 낮은 보수적 산업 구조
- 인천시·IPA·해수청 TF 운영으로 정책 지원 가능성

## 2. 프로젝트 개요

### 2.1 프로젝트 목적
기존의 파편화된 수출 프로세스(전화, 수기 장부, 엑셀)를 디지털화하여:
1. 딜러의 업무를 "실행"에서 "확인"으로 전환
2. 서류 처리 시간 90% 단축
3. 정산 투명성 확보 (6-Day 타임라인)
4. 바이어 직거래 차단 (블랙박스 전략)

### 2.2 솔루션 구성
- **AutoExport**: 문서 자동화 SaaS (B2B)
  - OCR 기반 서류 파싱
  - 자동 문서 생성 (Invoice, Packing List 등)
  - 차량별 서류 아카이브
  
- **Cariv**: D2G 글로벌 거래 플랫폼
  - AI 기반 자연어 검색
  - 경매/즉시구매 시스템
  - 해외 바이어용 웹 서비스

### 2.3 핵심 전략
- **수동적 워크플로우**: 딜러는 요청만 하고, 시스템이 자동 처리
- **블랙박스 전략**: 바이어 정보, 최종가 등 핵심 정보 마스킹
- **검차 필수**: 검차 완료 차량만 판매 가능

## 3. 시장 분석 및 환경

### 3.1 시장 규모
| 연도 | 수출 대수 | 수출 금액 | 전년 대비 |
|------|----------|----------|----------|
| 2019 | 46.9만 대 | 약 30억 달러 | - |
| 2023 | 63.9만 대 | 47.8억 달러 (6.3조원) | +57.8% |
| 2024 | 62.7만 대 | 약 50억 달러 (6.6조원) | -1.8% |
| 2025 (1~4월) | 29.7만 대 | 24.9억 달러 (3.4조원) | +52.4% (YoY) |

### 3.2 주요 수출국
1. **러시아** (13.4%) - 제재 우회, 가솔린 SUV 선호
2. **UAE** (6.8%) - 재수출 허브
3. **사우디아라비아** (3.2%) - 신흥 시장
4. **리비아** - 한국차 1위 수입국, 재수출
5. **키르기스스탄** - 러시아 우회 경로 (2021년 4,490대 → 2023년 50,905대, 11배 증가)

### 3.3 경쟁 환경
- **국내 B2C 플랫폼**: 엔카(53%), 케이카(21%), KB차차차(14%) - 수출 기능 미보유
- **수출 특화 플랫폼**: 오토위니(16개 언어), 수출로(20년 경험) - 문서 자동화 기능 미흡
- **차별화 포인트**: 문서 자동화 + 거래 플랫폼 통합

### 3.4 SWOT 분석

**강점 (Strengths)**
- 도메인 특화 AI 정확도
- TS 공단 API 연동 역량 (8억건+ 데이터 접근)
- 문서 자동화 + 거래 플랫폼 통합 설계
- 풀스택 팀 구성

**약점 (Weaknesses)**
- 상용 레퍼런스 및 매출 실적 부재
- UX/UI·백엔드 추가 인력 부족
- 보안·법률 인증 체계 미구축
- 글로벌 운영 경험 부족

**기회 (Opportunities)**
- 6조원+ 고성장 시장
- 산업 전반의 디지털 전환 니즈
- 인천시·IPA·해수청 TF 운영 → 정책 연계 가능
- 시리아·중동 전후 복구 수요 급증

**위협 (Threats)**
- 인천항 점유율 하락 (93% → 85%)
- 엔카·케이카 등 대형 플랫폼의 수출 기능 추가 가능성
- 대러시아 제재 강화 및 우회수출 단속
- 중국의 글로벌 중고차 수출 시장 진입

## 4. 비즈니스 모델

### 4.1 딜러 유입 동기
- **부가세 환급 9.09%**: 딜러의 핵심 유입 동기
- **서류 처리 자동화**: 25종 이상 서류를 자동 처리
- **빠른 자금 회전**: 6-Day 타임라인으로 정산 완료

### 4.2 수익 모델

**AutoExport (B2B SaaS)**
| 플랜 | 대상 | 월 요금 | 포함 기능 |
|------|------|---------|----------|
| Free Trial | 전체 | 30일 무료 | 차량 10대, 기본 파싱 |
| Starter | 영세 수출업자 | 99,000원 | 차량 50대, 기본 기능 |
| Professional | 중견 수출업자 | 299,000원 | 차량 무제한, 관세 연동 |
| Enterprise | 조합·대행업체 | 협의 | API, 화이트라벨 |

**Cariv (D2G 플랫폼)**
- 거래 수수료: 판매자 2%, 구매자 1%
- 프리미엄 리스팅: 월 50,000원~
- 구독 플랜: 해외 바이어 수수료 50% 할인

### 4.3 블랙박스 전략
- 바이어 연락처 마스킹
- 최종 수출가 마스킹 (딜러는 정산 확정가만 확인)
- 선적 상세 정보 마스킹
- 목적: 딜러의 바이어 직거래 시도 원천 차단

## 5. 타겟 사용자

### 5.1 Primary User: 딜러 (김철수, 45세)
**특성**
- 연령: 45세, 남성
- 지역: 인천 엠파크
- Tech Literacy: 낮음 (PC보다 모바일 선호)
- 성격: 성격이 급하고 IT 기기 조작에 서툼
- 말버릇: "그래서 내 돈 언제 들어와?"

**요구사항**
- 복잡한 건 딱 질색
- 직관적인 버튼과 확실한 알림 선호
- 빠른 자금 회전을 위한 효율적인 프로세스 필요

**Pain Points**
- 복잡한 서류 처리 (25종 이상)
- 불투명한 정산 ("깜깜이 정산")
- 수출업자의 '잠수'나 사기 위험

### 5.2 Secondary User: 해외 바이어
**특성**
- 글로벌 중고차 구매자
- 선호 차종: 현대 팰리세이드/싼타페, 기아 쏘렌토/모하비, 상용차 봉고/포터
- 결제 방식: L/C(신용장), T/T(전신환)

**요구사항**
- 신뢰할 수 있는 매물 탐색
- 투명한 거래 프로세스
- 다국어 지원 (16개 언어)

### 5.3 Tertiary User: 운영 매니저
**역할**
- 딜러 신청 건 검토
- 평가사 배정
- 수출 서류 오류 수정
- 바이어 매칭, 서류 생성, 정산 승인 처리

## 6. 핵심 기능 명세

### 6.1 딜러용 기능

#### 6.1.1 회원가입 및 인증
**목적**: 거래에 적합한 딜러만 시장에 진입시키기

**핵심 기능**
- 사업자 정보 입력 및 증빙 업로드
- 인증 상태(심사 중 / 승인 / 반려) 명확한 표시
- 인증 완료 후 자동 권한 부여
- 재인증 또는 수정 요청 기능

**목표치**
- 인증 완료율 높이되, 허위·부적합 딜러 유입률 최소화
- 인증 완료 이후 이탈률 최소화
- "한 번만 하면 끝난다"는 인식 형성

#### 6.1.2 차량 업로드 및 검차
**목적**: 차량을 플랫폼이 관리 가능한 자산 단위로 등록

**핵심 기능**
- 차량 기본 정보 입력 및 임시 저장
  - 차대번호(VIN), 차량번호, 제조사, 모델명, 연식, 주행거리
  - TS API 연동으로 자동 정보 채움
- 차대번호 기반 중복 등록 방지
- 검차 신청 (희망 일정 입력)
- 평가사 배정 (시스템 자동)
- 검차 결과 수정 불가 정책

**목표치**
- 검차 완료 차량 비율 증가
- 검차 이후 거래 성사율 유의미한 상승
- 상태 관련 클레임 감소

#### 6.1.3 판매 방식 선택
**일반 판매 방식**
- 딜러가 가격 설정 → 바이어가 가격 제안 → 딜러 수락 → 바이어 최종 구매 의사 재확인

**경매 방식**
- 설정된 기간 내 최고가 낙찰 시 강제 판매
- 빠른 회전과 가격 발견에 중점

**차량 상태 관리**
- 판매 완료, 판매 중 상태 설정 가능

#### 6.1.4 탁송 플로우
**조건**: 계약 체결 이후, 잔금 입금 확인 → 송금 확인 → 환전 확인 완료 후에만 탁송 신청 가능

**핵심 기능**
- 탁송 신청 조건 제어
- 일정 조율 및 배차 확정
- 차량 상태 동일 확인서 제출
- 서류·차키 인계 확인

**목표치**
- 거래 종료 이후 분쟁 최소화
- 탁송 지연 및 누락 방지
- "여기까지 오면 끝이다"라는 명확한 인식 제공

### 6.2 바이어용 기능

#### 6.2.1 회원가입
**개인 바이어**
- 기본 정보 (영문 이름, 생년월일, 국적, BL 수령용 주소지)
- 여권 정보 (여권번호, 만료일)
- 연락처 (국가, 전화번호, WhatsApp, Telegram)
- 동의 (이용약관, 개인정보 처리방침, 마케팅 동의)

**사업자 바이어**
- 회사 정보 (회사명, 사업자등록번호, 소재 국가, BL 수령용 주소)
- 담당자 정보 (담당자 영문명, 직책, 전화번호, 이메일)
- 회사 유효성 검증 (사업자등록증 업로드, 홈페이지)
- 연락처 (회사 공식 전화번호, WhatsApp, Telegram)
- 동의 (이용약관, 개인정보처리방침, 마케팅 동의)

#### 6.2.2 AI 검색
**자연어 검색**
- 사용자 입력 예시: "Show me low-mileage SUVs under $7,000."
- AI가 파싱하는 필드:
  - manufacturer (제조회사)
  - model (모델명)
  - year (연식)
  - mileage (키로수)
  - fuel_type (연료 타입)
  - country (나라)
  - price (가격)
- NULL 필드는 전체 범위로 자동 확장

**프롬프트 파싱 구조**
```
User Query → Prompt Template → LLM → JSON → Search Query Builder → 결과 리턴
```

#### 6.2.3 Quick Search
**필터 항목**
- 가격
- 키로수
- 연식
- 차종
- 연료 타입
- 브랜드
- 모델

#### 6.2.4 경매 시스템
**Auction (입찰 방식)**
- 시작가부터 여러 바이어가 입찰
- 종료 시 최고가가 낙찰
- 낙관적 락(optimistic)으로 동시성 보장

**Buy Now (즉시구매)**
- buy_now_price 이상을 제시하면 즉시 낙찰
- 첫 번째 성공 요청만 인정 (원자성 보장)

### 6.3 공통 기능

#### 6.3.1 Dashboard (딜러용)
**FEAT-001: Status 실시간 업무 현황판**

**FEAT-003: 긴급 알림 센터**

#### 6.3.2 Document Hub
**FEAT-007: Split-View 검증기 (OCR)**

**FEAT-008: 원클릭 서류 생성**
- 인보이스, 수출이행확인서 등 PDF 자동 생성

**FEAT-009: 디지털 서류 아카이브**

- 말소증/면장/BL 등 자동 분류 저장

| 탭(또는 영역) | 동일한 네비게이션 컨텍스트(GNB/LNB/Sidebar 기준 묶음) | 예: `딜러(대시보드)`, `바이어(홈)`, `Document Hub` |
| 1depths | 최상위 메뉴(그룹) | 예: 차량 관리, 판매 관리 |
| 2depths | 화면/리스트/페이지 단위 | 예: 차량 등록, 차량 목록 |

| contents | 사용자가 할 수 있는 일(기능 요약) + ID 참조 | `SCR-####`, `FLOW-##`, `FUNC-##`, `KPI-###`는 **contents에 포함** |

#### 6.4.2 행(로우) 작성 규칙 — (DOCX 친화)

- **1행 = “하나의 네비게이션 경로 + 그 경로에서 제공되는 핵심 행동(기능)”**을 표현한다.
- 병합 셀은 사용하지 않고, 미존재/미정은 `-`로 표기한다.
- contents는 1~2문장으로 끝내고, 상세 동작/예외 처리는 **6.5 매핑 표** 또는 Phase별 PRD(FRD)에서 확장한다.

### 6.5 기능↔화면↔플로우↔지표 매핑(관계 규칙) — 개발자 기준

#### 6.5.1 관계 규칙(정의)

- **FEAT(Feature)**: 사용자 가치 단위의 기능(에픽). 화면/흐름 관점으로 관리한다. (예: `FEAT-008 원클릭 서류 생성`)
- **FUNC(Function)**: 구현 단위(유저 액션/시스템 액션/정책 제어)로, 화면/플로우/KPI와 1차적으로 매핑한다. (예: `FUNC-60 파일 업로드`)
- **SCR(Screen)**: 사용자가 실제로 보는 화면/페이지/뷰. IA 표(6.4)에서 Depth로 표현된다.
- **FLOW(Flow)**: 사용자 여정의 단계적 흐름. 여러 SCR의 순서/상태 전이로 구성된다.
- **KPI**: 성과지표. 최소 1개 이상의 FEAT/FUNC에 연결되어야 한다.

#### 6.5.2 매핑 산정(추출) 규칙 — 문서 기반

- `기술설계 기능및 목표치.md`의 **“포함되어야 할 핵심 기능”** 항목은 원칙적으로 **FUNC로 승격**한다.
- 동일 문서의 **“목표치”** 항목은 가능한 한 **KPI 또는 FRD의 AC(달성기준)**로 변환한다.
- `중고차 거래 플랫폼 세부 플로우 및 BM 구조 .md`의 정책/철학 문장은 **제약 조건(정책) 또는 NFR**로 처리하고, 관련 FUNC/SCR에 연결한다.
- `기술설계서.md`의 화면/기능 설명은 **Buyer Front + Document Hub**의 SCR/FUNC로 분해해 레지스트리에 반영한다.

## 7. 기술 아키텍처

### 7.1 시스템 구성

**Dealer Front (딜러용 웹 클라이언트)**
- 기술 스택: React 18+, TypeScript, React Query, Zustand
- 배포: AWS CloudFront + S3 (정적 호스팅)
- 특징: SPA 구조, View Only 권한 관리

**Admin Back-office (관리자 대시보드)**
- 기술 스택: React Admin 또는 Next.js
- 기능: 바이어 매칭, 서류 생성, 정산 승인, 딜러 관리

**Core API Server**
- 기술 스택: Node.js (Express) 또는 Python (FastAPI)
- 데이터베이스: PostgreSQL (주 DB), Redis (캐싱)
- 서비스 구성:
  - Inspection Svc: TS 데이터 연동 및 검사 일정 스케줄링
  - Trade Svc: 경매/매칭 알고리즘 (블랙박스 처리)
  - Settlement Svc: 환율 계산, 수수료 차감, 정산서 생성
  - Docu Svc: OCR 파싱 및 PDF 생성
  - Auth Svc: 인증/인가 처리 (JWT 기반)
  - Notification Svc: 알림톡/푸시 발송

**Batch Server**
- 기술 스택: Node.js Cron 또는 Python Celery
- 기능: 정산 자동 계산, 알림 발송, 데이터 동기화

### 7.2 외부 인터페이스
- **TS 교통안전공단**: 차량 제원, 압류/저당 정보 실시간 조회
- **PG사**: 검사비 결제 및 1원 계좌 인증
- **OCR Engine**: 말소증 이미지 텍스트 추출 (Upstage 등)
- **Notification**: 카카오 알림톡 및 앱 푸시 전송

### 7.3 인프라 설계 (AWS)
**VPC 구성**
- Public Subnet: ALB, Bastion, Jenkins
- Private Subnet: RDS, S3, EC2 (Spring 서버)

**Security Group**
- cariv-sg-alb: 80, 443 (0.0.0.0/0)
- cariv-sg-app: 8080 (from sg-alb), 22 (from sg-bastion)
- cariv-sg-rds: 3306 (from sg-app)

**CI/CD**
- Jenkins 기반 자동 배포
- GitHub Webhook 연동
- ECR을 통한 Docker 이미지 관리

## 8. 사용자 플로우

### 8.1 딜러 플로우 (8단계 라이프사이클)

**단계 0: 시작**
- 딜러 로그인

**단계 1: 회원가입 및 인증**
- 휴대폰 번호, 이름 (PASS 인증)
- 약관 동의, 계정 생성
- 출력: GUEST 등급 부여

**단계 2: 사업자 인증**
- 사업자 인증 완료 여부 확인 (Y/N)
- N → 사업자등록증 이미지 업로드 → OCR 처리 및 진위 확인 → DEALER 등급으로 전환

**단계 3: 검사 신청**
- 차량번호 입력, 방문일정 선택
- TS API로 기본정보 자동 조회
- 출력: Draft 상태 생성

**단계 4: 평가사 배정**
- 시스템 자동 배정
- 출력: Inspecting 상태로 전이

**단계 5: 검사 수행**
- 평가사가 사진 촬영, 성능 점검, 옵션 체크
- 출력: 검사 리포트 업로드

**단계 6: 글로벌 마켓 노출**
- 시스템 자동 처리
- Cariv API 연동, 검색 인덱싱 생성
- 출력: Active 상태로 전이

**단계 7: 매칭/입찰**
- 시스템 자동 처리
- 해외 바이어 조회/질문/제안 수집, 매칭 알고리즘 실행
- 출력: Bidding 상태로 전이

**단계 8: 바이어 제안 수락**
- 바이어 제안 수락 여부 확인 (Y/N)
- Y → 계약 확정 및 서류 생성 → Locked 상태로 전이
- N → Bidding 상태 유지 (계속 매칭 대기)

**단계 9: 말소증 업로드**
- 자동차말소등록사실증명서 이미지 업로드
- OCR 처리 및 검증
- 출력: 말소증 파일 저장

**단계 10: 바이어 결제 확인**
- 시스템 자동 처리
- 바이어 입금 확인
- 출력: Settlement 상태로 전이

**단계 11: 정산 계산**
- 시스템 자동 처리
- 매입 확정가 - 수수료 + 검사비 환급 계산
- 출력: 정산 내역서 생성

**단계 12: 인계 승인**
- 딜러가 6자리 PIN 번호 입력
- 책임 이관 타임스탬프 기록
- 출력: 인계 완료 확인

**단계 13: 정산 송금**
- 시스템 자동 처리
- 딜러 계좌 입금 처리
- 출력: Sold 상태로 전이

**단계 14: 거래 완료**
- 거래 완료 알림

## 9. 비기능 요구사항

### 9.1 UI/UX 가이드라인
- **Theme**: Bloomberg Style Dark Mode (#121212 배경)
- **Density**: 스크롤을 최소화하고 한 화면에 정보를 집약적으로 배치
- **Visual Cue**: 
  - 긍정적 정보(수익, 입금): Neon Green (#00E096)
  - 부정/경고(반려, 지연): Neon Red (#FF4B4B)
- **Interaction**: 클릭 횟수를 최소화, 스크롤보다는 탭(Tab) 구조 활용

### 9.2 성능 및 안정성
- **Response Time**: 
  - 대시보드 로딩 < 1.5초
  - OCR 처리 < 3초
- **Concurrency**: 동일 차량에 대한 중복 계약 방지 (Optimistic Lock 적용)
- **Availability**: AWS Multi-AZ 구성으로 가용성 99.9% 보장

### 9.3 보안 및 규정 준수
- **Encryption**: 개인정보(주민번호, 여권, 계좌) DB 저장 시 AES-256 암호화
- **Audit Log**: 딜러의 모든 중요 액션(인계 확인, 출금 신청)에 대한 IP 및 시간 로그 보관
- **컴플라이언스**: 
  - 대러시아 제재 체크 (2000cc↑, 5만$↑)
  - 수출 대상국 스크리닝

### 9.4 데이터 관리
- **백업**: 정기 백업 및 복구 테스트
- **보관**: 감사 로그는 WORM(Write Once Read Many) 스토리지에 별도 보관
- **접근 제어**: 역할 기반 접근 제어 (RBAC)

- Frontend Developer: 1명 (해외 바이어 웹 UI, 반응형 화면, 검색/경매 페이지)
- Designer: 1명 (웹 UX/UI, 가입 화면, 메인 구조, 브랜드 가이드)

**문서 자동화 프로그램 개발팀 – 총 3명**
- Backend Developer: 1명 (OCR Pipeline, 문서 Parsing, 변환 엔진)
- Frontend Developer: 1명 (파일 업로드 UI, Dashboard, DMS 화면)
- Designer: 1명 (문서 분석 UI/흐름, 편집기 UX, 관리자 대시보드)

### 10.4 세부 태스크 플랜(통합 매핑 기반) — 보고용

본 태스크 플랜은 **6.5 기능 매핑 레지스트리(FUNC/SCR/FLOW/KPI)**를 기준으로 작성한다. “한 기능(FUNC)”을 구현 가능한 단위로 쪼개어 FE/BE/데이터/QA 산출물을 명시한다.

#### 10.4.1 태스크 산정 규칙(개발자 기준)

- **FUNC 1개당 최소 1개의 FE 태스크 + 1개의 BE 태스크**를 생성한다. (시스템 자동/정책 기능은 BE 중심)
- 각 태스크는 **완료 기준(AC/검증 방법)**을 포함한다.
- KPI가 연결된 기능은 **측정 이벤트(로그/메트릭) 태스크를 추가**한다.

## 11. 리스크 관리

### 11.1 운영 리스크
**리스크**: 탁송 중 차량 파손 또는 사고 발생

**완화 전략**
- 인계 확인 시점을 명확히 정의하여 책임 소재 명문화
- 탁송 보험 가입을 의무화
- 인계 시점 사진 촬영을 의무화

### 11.2 금융 리스크
**리스크**: 해외 바이어 송금 지연으로 인한 정산 지연

**완화 전략**
- 6일 타임라인에 여유 버퍼 포함
- 플랫폼 자체 자금으로 선정산 옵션 검토
- 바이어 신용도 평가 시스템 구축

### 11.3 법적 리스크
**리스크**: 수출 말소 불이행 또는 대포차 유용

**완화 전략**
- 딜러에게 말소증 업로드 강제 및 미이행 시 페널티 부과
- 딜러 인증(사원증) 강화
- 말소증 OCR 검증 및 자동화

### 11.4 이탈 리스크
**리스크**: 딜러가 바이어와 직거래 시도

**완화 전략**
- 모든 바이어 정보를 마스킹 및 블랙박스화
- 적발 시 영구 제명 조항을 약관에 삽입
- 바이어 연락처 추적 시스템 구축

### 11.5 기술 리스크
**리스크**: 외부 API 장애로 인한 서비스 중단

**완화 전략**
- 외부 API 다중화(Fallback API 준비)
- 캐싱 전략으로 API 호출 최소화
- 재시도 정책 및 에러 핸들링 강화

### 11.6 데이터 리스크
**리스크**: 개인정보 유출 또는 데이터 손실

**완화 전략**
- AES-256 암호화 적용
- 정기 백업 및 복구 테스트
- 접근 로그 모니터링 및 이상 징후 탐지

## 12. 성공 지표 (KPI)

### 12.1 딜러 관련 KPI

**KPI-001: 전환율 (Conversion)**
- 목표: 가입 딜러 대비 '검사 신청(매물 등록)' 전환율 30% 이상
- 측정 주기: 월간
- 목표 달성 기준: 첫 달 20%, 3개월 내 30% 달성

**KPI-002: 재방문율 (Retention)**
- 목표: 재방문율 80% 이상
- 측정 주기: 월간
- 목표 달성 기준: 첫 달 60%, 6개월 내 80% 달성

**KPI-003: 효율성 (Efficiency)**
- 목표: 딜러 1인당 매물 등록 소요 시간 1분 미만 (기존 20분 → 1분)
- 측정 주기: 주간
- 목표 달성 기준: 평균 1분 이하, 95% 사용자가 2분 이내 완료

**KPI-004: 정산 속도 (Settlement Speed)**
- 목표: 바이어 결제 확인부터 딜러 계좌 입금까지 평균 6일 이내
- 측정 주기: 주간
- 목표 달성 기준: 평균 6일 이하, 90% 거래가 7일 이내 완료

**KPI-005: 블랙박스 효과 (Blackbox Effectiveness)**
- 목표: 딜러의 바이어 직거래 시도율 5% 미만
- 측정 주기: 월간
- 목표 달성 기준: 첫 달 10% 이하, 6개월 내 5% 이하

### 12.2 바이어 관련 KPI

**KPI-006: 가입 전환율**
- 목표: 웹사이트 방문자 대비 회원가입 전환율 5% 이상

**KPI-007: 검색 만족도**
- 목표: AI 검색 결과 만족도 70% 이상

**KPI-008: 거래 성사율**
- 목표: 차량 조회 대비 거래 성사율 10% 이상

### 12.3 플랫폼 관련 KPI

**KPI-009: 월간 활성 사용자 (MAU)**
- 목표: 첫 달 100명, 6개월 내 1,000명

**KPI-010: 거래 건수**
- 목표: 첫 달 10건, 6개월 내 100건

**KPI-011: 수익**
- 목표: 첫 달 100만원, 6개월 내 1,000만원

**핵심 용어**
- **딜러 (Dealer)**: 중고차 매매업을 하는 판매자

- **Bidding**: 매칭/입찰
- **Locked**: 계약/서류
- **Settlement**: 정산/종료
